# Relay TLS 与统一证书简化交互设计

- 日期：2026-04-07
- 状态：已确认设计，待进入实现计划
- 范围：`panel/frontend`、`panel/backend`、现有 Relay/证书数据模型的兼容式演进

## 1. 背景

当前 Relay 监听器 TLS 配置同时暴露了以下概念：

- 监听器自己对外出示的证书
- 下一跳监听器证书的信任方式
- TLS 校验策略（Pin / CA / 组合）
- 自签名证书例外

这些概念都真实存在，但当前 UI 直接把底层字段暴露给用户，导致普通用户难以理解“谁校验谁”“Pin 校验的是哪张证书”“监听证书和可信 CA 的区别”。

同时，“统一证书”页虽然已有 `certificate_type=uploaded` 的概念，但没有真正的手动上传入口，导致用户无法完成最基础的 PEM 导入流程。

## 2. 目标

### 2.1 主要目标

1. 将 Relay 监听器默认配置简化为“普通用户可理解、可完成”的表单。
2. 保留现有高级 TLS 能力，不破坏兼容性。
3. 将“统一证书”变成真正的单一证书入口。
4. 补齐手动上传 PEM 证书的最小可用能力。

### 2.2 非目标

本次不引入：

- mTLS / 客户端证书校验
- PFX / PKCS#12 上传
- 大规模重构证书存储模型
- 对 Go agent TLS 运行时语义做破坏式修改

## 3. 现状语义澄清

### 3.1 Relay TLS 方向

当前实现中，发起连接的一方会校验下一跳 Relay 监听器出示的服务端证书：

- A 连 B：A 校验 B 的监听器证书
- B 连 C：B 校验 C 的监听器证书

当前不支持 mTLS，因此监听器不会反向校验来访方客户端证书。

### 3.2 关键字段语义

- `certificate_id`：本 Relay 监听器对外出示的服务端证书
- `pin_set`：下一跳监听器证书公钥指纹（SPKI SHA-256）白名单
- `trusted_ca_certificate_ids`：用于校验下一跳监听器证书链的 CA 集合
- `allow_self_signed`：允许信任自签名的下一跳证书，但不是跳过校验
- `tls_mode`：底层验证组合策略，当前支持：
  - `pin_only`
  - `ca_only`
  - `pin_or_ca`
  - `pin_and_ca`

## 4. 总体方案

采用“默认自动、保留高级设置”的方案。

### 4.1 统一证书

保留单一“统一证书”页面，不拆分成多个页面；但在新建证书时引入“用途模板”，让用户用业务语言进入对应流程。

### 4.2 Relay 监听器

Relay 监听器默认仅展示必需字段和“自动信任策略”；高级 TLS 字段隐藏到折叠区域中，仅在需要时使用。

### 4.3 兼容原则

- 底层数据结构保持兼容
- 后端与 agent 继续接受现有 `tls_mode` / `pin_set` / `trusted_ca_certificate_ids`
- 新 UI 只是在默认场景下自动推导这些字段

## 5. 统一证书页设计

## 5.1 页面定位

“统一证书”继续作为全局证书中心，负责：

- 查看现有证书
- 新建不同用途证书
- 编辑证书元信息
- 手动上传 PEM 材料

## 5.2 新建流程：用途模板

用户点击“新建证书”后，先选择用途模板：

1. 网站 HTTPS
2. Relay 监听证书
3. Relay CA 证书
4. 手动上传证书
5. 内部自签证书

模板作用是预填默认值，而不是创建独立的数据模型。

## 5.3 模板默认映射

### 网站 HTTPS

- `usage=https`
- 默认 `certificate_type=acme`
- 默认按现有签发能力展示签发方式

### Relay 监听证书

- `usage=relay_tunnel`
- 支持自动签发 / 手动上传 / 内部自签

### Relay CA 证书

- `usage=relay_ca`
- 优先用于“签发或信任 Relay 内部链路”

### 手动上传证书

- `certificate_type=uploaded`
- 直接进入 PEM 粘贴表单

### 内部自签证书

- 默认 `self_signed=true`
- 用于内网/Relay 等非公网 ACME 场景

## 5.4 手动上传入口

当证书来源为“手动上传”时，显示 PEM 输入区。

最小可用版本支持文本粘贴：

- 证书链 PEM
- 私钥 PEM
- CA PEM（可选）

第一版不要求必须提供本地文件上传控件；如果前端额外提供“读取文件到文本框”的辅助能力，应仍以 PEM 文本为最终提交内容。

## 5.5 手动上传验证规则

### 服务端证书（例如 HTTPS / Relay 监听）

- 证书链 PEM 必填
- 私钥 PEM 必填
- 证书与私钥必须匹配

### CA 证书

- CA PEM 必填
- 可不要求私钥，除非后续扩展到“导入可签发内部 CA”

### 通用规则

- PEM 格式必须合法
- 非法内容返回明确错误信息

## 5.6 列表展示

统一证书列表增加或强调以下信息：

- 名称/域名
- 用途
- 来源（自动签发 / 手动上传 / 内部 CA）
- 是否自签
- 当前状态
- 最近签发/导入时间

## 6. Relay 监听器设计

## 6.1 默认表单字段

普通用户默认看到：

- 名称
- 监听地址
- 监听端口
- 本监听器对外证书
- 信任策略
- 启用监听器

不默认显示：

- TLS 模式
- Pin Set
- 可信 CA 证书
- 允许自签名

这些内容移到“高级设置”折叠区域。

## 6.2 文案重命名

### 监听证书

改为：**本监听器对外证书**

### 可信 CA 证书

改为：**用于校验下一跳的 CA 证书**

### Pin Set

改为：**用于校验下一跳的证书指纹（Pin）**

### 允许上游使用自签名证书

改为：**允许信任自签名的下一跳证书**

这样可避免“上游/下游/本监听器”混用造成歧义。

## 6.3 信任策略

默认展示的“信任策略”选项：

1. 自动（推荐）
2. 仅信任指定证书
3. 仅信任指定 CA
4. 高级设置

其中，“高级设置”不是值，而是进入更底层配置的方式。

## 6.4 自动模式的推导规则

自动模式不直接要求用户理解 `tls_mode`，而是根据实际填写内容推导：

- 只有 Pin → `pin_only`
- 只有 CA → `ca_only`
- 同时存在 Pin 和 CA → `pin_and_ca`
- Pin 和 CA 都为空 → 表单报错

设计上默认选 `pin_and_ca` 而不是 `pin_or_ca`，因为它更安全，也更符合生产默认值。

`pin_or_ca` 仍保留在高级设置中，供迁移或兼容场景使用。

## 6.5 高级设置

展开后显示：

- TLS 模式（四种全部显示）
- Pin Set
- 可信 CA 证书
- 允许信任自签名的下一跳证书

高级设置用于：

- 兼容旧配置
- 显式选择 `pin_or_ca`
- 精细排障
- 资深用户的安全策略控制

## 6.6 证书必填约束

当前 agent 运行时要求：启用的 Relay 监听器必须有 `certificate_id`。

因此 UI 应同步为：

- 启用监听器时，“本监听器对外证书”必填
- 删除“启用但不绑定证书”的错误心智

## 7. 数据映射与兼容性

## 7.1 Relay 监听器

前端默认模式只是映射到现有字段：

- `certificate_id`
- `tls_mode`
- `pin_set`
- `trusted_ca_certificate_ids`
- `allow_self_signed`

不修改 agent 运行时含义。

## 7.2 统一证书

用途模板只是预填：

- `usage`
- `certificate_type`
- `self_signed`
- `scope`
- `issuer_mode`

不引入新的必须存储字段，除非实现手动上传时需要新增证书材料存储位置。

## 7.3 兼容已有记录

已有证书和 Relay 监听器记录必须可继续编辑：

- 旧记录进入新 UI 时，应根据现有底层字段反推默认显示状态
- 若记录使用 `pin_and_ca`，前端必须正确显示，不能丢失
- 若记录使用 `pin_or_ca`，前端应进入高级模式并明确标注

## 8. API / 后端设计要求

## 8.1 统一证书

后端需要支持“手动上传证书”所需的数据接收与验证逻辑。

第一版可以采用：

- 在新建/更新证书接口中接收 PEM 文本字段
- 服务端完成解析、校验、持久化

### 建议最小字段

- `certificate_pem`
- `private_key_pem`
- `ca_pem`

字段命名可在实现阶段最终确定，但应避免和现有证书元字段冲突。

## 8.2 Relay 监听器

后端继续接受现有 payload，不需要破坏式修改接口。

若前端采用“自动模式”，则由前端在提交前完成底层字段推导；后端仍按既有验证执行。

## 8.3 验证补齐

后端/前端需保持一致：

- 启用的 Relay 监听器必须绑定证书
- `pin_and_ca` 必须同时具备 Pin 与 CA
- `pin_only` 必须有 Pin
- `ca_only` 必须有 CA
- 自动模式下不得提交“Pin/CA 都为空”

## 9. 用户流程

## 9.1 创建 Relay 监听器（推荐路径）

1. 用户新建 Relay 监听器
2. 填写名称、地址、端口
3. 选择“本监听器对外证书”
4. 若没有证书，点击新建证书
5. 在统一证书中选择“Relay 监听证书”或“手动上传证书”
6. 返回 Relay 监听器表单
7. 信任策略保持“自动”
8. 保存

## 9.2 上传已有证书

1. 进入统一证书
2. 选择“手动上传证书”
3. 粘贴 PEM 文本
4. 保存
5. 在 HTTPS 或 Relay 监听器中引用

## 10. 风险与处理

### 10.1 风险：自动模式与现有底层配置理解不一致

处理：

- 明确自动模式的推导规则
- 对老记录提供高级模式回显

### 10.2 风险：手动上传引入证书材料校验复杂度

处理：

- 第一版仅支持 PEM
- 强制最小校验：格式合法、证书/私钥匹配

### 10.3 风险：UI 简化后高级用户担心能力被移除

处理：

- 保留高级设置
- 明确标注“高级 TLS 设置”

## 11. 实现边界建议

本设计建议拆分为以下实现块：

1. 统一证书：用途模板 UI
2. 统一证书：手动上传 PEM
3. Relay 监听器：默认简化表单
4. Relay 监听器：高级设置与现有数据兼容映射
5. 后端：证书材料接收/校验/存储

## 12. 验收标准

### 12.1 普通用户

- 不理解 `tls_mode` 也能创建可工作的 Relay 监听器
- 不理解 Pin / CA 细节也能完成推荐配置
- 可以在统一证书中完成手动上传 PEM

### 12.2 高级用户

- 可以继续编辑 `pin_only` / `ca_only` / `pin_or_ca` / `pin_and_ca`
- 不会因为新 UI 丢失旧配置语义

### 12.3 兼容性

- 已有 Relay 监听器和证书记录能够正常展示、编辑、保存
- agent 现有 TLS 运行时行为不被破坏
