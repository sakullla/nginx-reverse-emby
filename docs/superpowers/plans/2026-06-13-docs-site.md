> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a VitePress documentation website under `docs-site/` and publish it through GitHub Pages without repurposing the root `docs/` directory.

**Architecture:** The documentation site is an independent Node/VitePress package in `docs-site/`. GitHub Actions builds only that package and deploys `docs-site/.vitepress/dist` to Pages. Screenshots for future docs live under `docs-site/public/screenshots/` and can be captured with Chrome MCP.

**Tech Stack:** VitePress, Node.js, Markdown, GitHub Pages Actions.

```md
# Getting Started

Nginx-Reverse-Emby is a pure Go control-plane and agent runtime for managing reverse proxy rules across local and remote nodes.

The recommended deployment path is Docker Compose. It starts the control panel, API, storage, and embedded local agent in one packaged stack.

## Requirements

- Docker with Compose support.
- A server that can expose the panel port.
- A strong `API_TOKEN` for panel login.
- A `MASTER_REGISTER_TOKEN` if remote agents will join the master.

## Quick Path

```bash
mkdir -p nginx-reverse-emby && cd nginx-reverse-emby
curl -O https://raw.githubusercontent.com/sakullla/nginx-reverse-emby/main/docker-compose.yaml
mkdir -p data
```

Edit `docker-compose.yaml`, set `API_TOKEN`, and start the stack:

```bash
docker compose up -d
```

Open `http://<server-ip>:8080` and log in with `API_TOKEN`.

## Next Steps

- Continue with [Docker Compose](./docker-compose.md) for the full deployment notes.
- Read [Agent Setup](./agent.md) before adding remote nodes.
- Review [Environment Variables](/reference/environment.md) before production deployment.
```

# Docker Compose

Docker Compose is the recommended runtime for most deployments.

## Data Directory

The compose file persists panel state, SQLite data, certificates, and runtime files to `./data` on the host.

Do not commit files from this directory.

## Required Environment

```yaml
environment:
  API_TOKEN: your-secure-token
  MASTER_REGISTER_TOKEN: your-register-token
  NRE_TIMEZONE: Asia/Shanghai
```

`API_TOKEN` is required for panel access. `MASTER_REGISTER_TOKEN` is used when agents register with the master.

## Host Network Mode

The default Compose file uses host networking because proxy listeners are configured dynamically from the panel. Docker bridge networking cannot publish new listening ports after the container has already started.
```

# Agent Setup

Agents execute proxy rules on local or remote nodes. They pull desired state from the master through heartbeat sync.

## Linux

```bash
curl -fsSL http://master.example.com:8080/panel-api/public/join-agent.sh | sh -s -- \
  --register-token your-register-token \
  --install-systemd
```

## macOS

```bash
curl -fsSL http://master.example.com:8080/panel-api/public/join-agent.sh | sh -s -- \
  --register-token your-register-token \
  --install-launchd
```

## NAT Nodes

NAT agents only need outbound access to the master. The master does not need to open an inbound connection to the agent.

## Windows

Windows nodes are expected to use the client package distributed through GitHub Releases.
```

# Environment Variables

This page lists the core configuration variables used by the control plane and agents.

## Control Plane

| Variable | Default | Description |
| --- | --- | --- |
| `API_TOKEN` | required | Panel login token and API credential. |
| `MASTER_REGISTER_TOKEN` | same as `API_TOKEN` | Token used by agents during registration. |
| `PANEL_BACKEND_HOST` | `0.0.0.0` | Control-plane bind address. |
| `PANEL_BACKEND_PORT` | `8080` | Control-plane bind port. |
| `NRE_ENABLE_LOCAL_AGENT` | `1` | Enables the embedded local agent. |
| `NRE_DATABASE_DRIVER` | `sqlite` | Database driver: `sqlite`, `postgres`, or `mysql`. |
| `NRE_DATABASE_DSN` | empty | Database DSN. SQLite defaults to `NRE_DATA_DIR/panel.db`. |
| `NRE_TIMEZONE` | `UTC` | IANA timezone for panel aggregation boundaries. |

## Agent

| Variable | Description |
| --- | --- |
| `NRE_AGENT_ID` | Agent identifier. |
| `NRE_AGENT_NAME` | Agent display name. |
| `NRE_AGENT_TOKEN` | Agent authentication token. |
| `NRE_MASTER_URL` | Master control-plane URL. |
| `NRE_DATA_DIR` | Agent data directory. |
```

# Certificates

The control plane can manage ACME certificates and certificate material for proxy rules.

## HTTP Validation

HTTP validation requires port `80` to be reachable from the public internet.

## DNS Validation

DNS validation is recommended when the proxy cannot expose port `80` or when wildcard certificates are needed.

For Cloudflare, configure the DNS provider and token through environment variables:

```env
ACME_DNS_PROVIDER=cf
CF_TOKEN=your-cloudflare-api-token
```

Never commit DNS tokens or private keys.
```

# WireGuard

WireGuard profiles are managed from the control panel and can be used by Relay or L4 rules.

Profiles include keys, listen addresses, interface addresses, peers, DNS, MTU, enabled state, and tags.

Sensitive fields are redacted by the API and panel. Leaving a redacted value unchanged during editing preserves the stored secret.

# Migration

The control plane includes storage migration support for moving from SQLite to PostgreSQL.

Run the migration command from an environment that can read the source SQLite file and connect to the destination database:

```bash
nre-control-plane migrate-storage \
  --from-driver sqlite \
  --from-dsn /opt/nginx-reverse-emby/panel/data/panel.db \
  --to-driver postgres \
  --to-dsn 'postgres://nre:nre@127.0.0.1:5432/nre?sslmode=disable'
```

When certificate material is stored outside the inferred SQLite data root, pass explicit data root flags.
```

# FAQ

## Why does the default Compose file use host networking?

The embedded local agent can create proxy listeners dynamically. Host networking lets those listeners bind directly on the host without changing container port mappings.

## Where is panel data stored?

In the default Docker Compose deployment, panel data is stored in `./data` on the host and mounted into the container.

## Can I keep using the root docs directory for temporary notes?

Yes. The public documentation site source lives in `docs-site/`, not the root `docs/` directory.

- [ ] **Step 1: Create `.github/workflows/docs-pages.yml`**

Use this exact content:

```yaml
name: Docs Pages

on:
  push:
    branches:
      - main
    paths:
      - docs-site/**
      - .github/workflows/docs-pages.yml
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: npm
          cache-dependency-path: docs-site/package-lock.json

      - name: Setup Pages
        uses: actions/configure-pages@v5

      - name: Install dependencies
        working-directory: docs-site
        run: npm ci

      - name: Build docs
        working-directory: docs-site
        run: npm run build

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: docs-site/.vitepress/dist

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```
