## 1. 问题陈述
- 问题句：我们要怎样让用户拿到一个记录 ID 后，在任何搜索入口都能立刻定位到它，而不被「当前所在服务器」卡住？
- 当前问题：
  - 规则/L4 的 id 是**按服务器各自编号**的（每个 agent 的规则从各自最大 id 往后排，且 HTTP/L4 共享同一套 id 空间），即 id 在服务器间**非全局唯一**。
  - 功能页搜索框（规则/L4/证书/节点页）只在「当前 agent 已加载的数据」上做纯前端过滤，输入别的服务器的 id 直接搜不到。
  - 全局搜索虽跨服务器拉取数据，但 `#id=` 仅在「点击结果后」作为预填语法带过去，全局框内直接输入 `#id=` 不会被识别为 id 过滤。
- 受影响用户 / 角色：面板运维 / 管理员（跨多服务器管理 HTTP规则、L4规则、证书、节点、relay 监听器的用户）。
- 为什么现在要做：从日志 / 工单 / 同事拿到一个记录 ID 后，得先知道它在哪个服务器、手动切换到该服务器才能搜到，定位路径断裂、体验差。
- 现有替代方案 / 当前做法：手动判断目标服务器 → 切换到该服务器 → 在功能页搜索 → 定位；或全局搜索后从结果列表里肉眼找。

## 2. 推荐方向
- 推荐方向：按场景区分 `#id=` 的处理——功能页搜索框用于精准定位（自动跳转 + 定位），全局搜索用于扫描（结果列表点选）。
- 选择理由：两个入口的用户心智不同——功能页是「我就要定位这一条」（跳转最高效），全局 Ctrl+K 是「搜索/扫描」（列表更符合预期，且不破坏现有面板交互习惯）。
- 备选方向与取舍：
  - 方向 A（两处统一走结果列表）：交互一致，但功能页多一步点选、降低「直达」效率。未选。
  - 方向 B（两处统一自动跳转）：功能页合适，但全局搜索单命中也跳转会破坏 Ctrl+K 的扫描式体验。未选。
- 不选其它方向的原因：见方向 A/B 取舍。

## 3. 目标与成功标准
- 目标：`#id=` 成为「有 ID 就直达」的能力，跨服务器解析，不被当前服务器作用域限制。
- 成功标准：在 HTTP规则 / L4规则 / 证书 / 节点页搜索框、以及全局搜索框中输入 `#id=xxx`，均可正确跨服务器解析（见验收 A1–A6）。
- 验收标准：
  - A1（功能页·当前命中）：在某 agent 的功能页搜索框输入 `#id=`，若当前 agent 存在该 id，则停留、列表过滤到该行、滚动到可见、短暂高亮。
  - A2（功能页·跨服务器单命中）：当前 agent 无该 id、仅另一个服务器有，则跳转到该服务器对应页面并定位（过滤/滚动/高亮）。
  - A3（功能页·跨服务器多候选）：当前 agent 无该 id、多个服务器有，则列出候选（标注来源服务器）供用户选择，选择后跳转并定位。
  - A4（全局·列表）：全局搜索框输入 `#id=` 后，结果列表正确显示跨所有服务器的 id 命中项，由用户点选；不自动跳转（含单命中）。
  - A5（全局·普通关键词不变）：全局搜索普通关键词行为与现状一致，结果列表的多类型混排不变。
  - A6（语法兼容与不做项）：`#id=` 保持精确匹配，不做前缀/模糊；定位不自动打开详情/编辑。
- 验收与需求锚点映射：A1↔R1+R2+R8；A2↔R1+R3+R8；A3↔R1+R4；A4↔R5；A5↔R6；A6↔R7+R8。
- 度量 / 可观察结果：跨服务器场景下 `#id=` 命中并定位/列出的成功率；功能页单命中跳转的耗时。

## 4. 范围
- 最小可交付范围 / 本次包含：
  - 功能页搜索框（HTTP规则、L4规则、证书、节点页）：`#id=` 跨服务器解析 + 当前优先 + 单命中跳转定位 + 多候选列表。
  - 全局搜索（Ctrl+K）：`#id=` 识别为 id 过滤、跨服务器显示结果列表、用户点选。
  - 跳转后定位（过滤 + 滚动 + 高亮）。
- 不包含：
  - `#id=` 前缀 / 模糊 / 范围匹配。
  - 自动打开详情 / 编辑抽屉。
  - 后端 id 分配模型改造（id 仍按 agent 编号）。
  - 全局搜索多类型结果列表本身的形态改造（仅让 `#id=` 在其中正确过滤）。
- 关键约束：
  - id 按服务器各自编号、HTTP/L4 共享 id 空间 → 必须处理多命中。
  - 保持现有 `#id=` 精确匹配语法（`/^#id=(\S+)$/`）兼容。
  - 功能页现为纯前端单 agent 过滤、全局为后端 API 跨 agent → 跨服务器解析需新增逻辑。
  - 涉及记录类型与各搜索现有覆盖一致：HTTP规则、L4规则、证书、节点、relay 监听器。
- 依赖与前置条件：全局搜索现有 `fetchAllAgentsRules / fetchAllAgentsL4Rules / fetchAllAgentsCertificates / fetchAllAgentsRelayListeners` 跨 agent 拉取能力；功能页需具备跨 agent 解析/查询手段（待技术方案确认走前端复用还是新增后端 API）。

## 5. 业务规则与交互要求
- 核心业务能力：`#id=` 跨服务器直达定位（功能页跳转、全局列表）。
- 关键业务规则：
  - 解析范围：`#id=` 始终跨所有服务器解析，不受当前 `selectedAgentId` / `route.query.agentId` 限制。
  - 优先级（功能页）：当前服务器优先；当前无 → 跨服务器；跨服务器唯一 → 跳转；跨服务器多个 → 候选列表。
  - 全局搜索：始终走结果列表，不做自动跳转。
  - 匹配语义：精确等值匹配（`/^#id=(\S+)$/`），不含前缀/模糊。
- 关键数据 / 信息：记录 id（按 agent 编号）、agentId（来源服务器）、记录类型、列表渲染数据。
- 用户操作与反馈：输入 `#id=` → 解析 → 功能页跳转/定位或候选列表；全局列表点选。
- 权限与数据范围：遵循现有面板权限；跨服务器读取仍受现有可见性约束。
- 状态 / 流程变化：功能页跳转会切换上下文（见 R9）。
- 审计、导入导出、历史记录等特殊要求：无。

## 6. 边界与异常
- 异常场景：`#id=` 在所有服务器均无命中 → 友好空状态/提示，不跳转、不报错。
- 边界场景：跨服务器多候选；id 含特殊字符（受现有正则约束）；当前 agent 与其它服务器同时命中同一 id（按「当前优先」停留）。
- 空数据 / 重复数据 / 无权限：空列表提示；重复 id（多服务器同 id）走候选列表；无权限的服务器不在候选/结果中显示。
- 兼容与历史数据：现有 `#id=` 语法与点击结果带 `#id=` 跳转的行为保持兼容；不改变历史数据。

## 7. 假设与验证
- 当前假设：
  - 「但功能搜索」= 功能页内搜索框；「全部搜索」= 全局搜索 GlobalSearch。
  - `#id=` 中的 id 指记录的 id 字段（HTTP/L4 规则、证书、节点等的 id）。
  - 跳转切换全局 `selectedAgentId` 与现有全局搜索跳转一致（R9）。
- 关键假设待验证：
  - [ ] 假设：跳转切换全局 `selectedAgentId` 不引发其它页面副作用问题；验证方式：研发实现时核对 AgentContext 与路由 query 同步的影响面。
  - [ ] 假设：跨服务器 id 解析可复用全局搜索现有 `fetchAllAgents*` 能力或需新增后端 API；验证方式：技术方案阶段评估性能与实现成本。
- 风险：跨服务器解析若走「前端拉取所有 agent」可能在大规模 agent 下有性能/数据量问题；需技术方案评估。
- 待补齐：跨服务器解析的前后端实现路径（前端复用 vs 新增后端 API）。

## 8. 不做事项
- `#id=` 前缀 / 模糊 / 范围匹配：保持精确匹配即可，避免放大范围。
- 自动打开详情 / 编辑抽屉：定位到行即可，避免打断。
- 后端 id 分配模型改造：本次不改变 id 按 agent 编号的现状。
- 全局搜索结果列表形态重设计：仅让 `#id=` 在其中正确过滤，不改列表本身。
