# HTTP 反向代理

本文将带你完成第一个 HTTP 反向代理规则的配置。假设你已完成[部署](./deploy.md)，并且能正常打开面板。

## 什么是 HTTP 反向代理？

简单来说，就是让一个域名指向你的服务。比如：

```text
用户访问: https://app.example.com
实际代理到: http://192.168.1.100:8096
```

用户只需要记住一个域名，nginx-reverse-emby 会帮他们把请求转发到实际的服务器上。这对 Emby、Jellyfin 或其他 Web 服务都适用。

## 1. 登录面板

在浏览器打开：

```text
http://<服务器 IP>:8080
```

输入你在 `docker-compose.yaml` 中设置的 `API_TOKEN` 登录。

![登录面板](/screenshots/panel-login.png)

---

## 2. 进入 HTTP 规则页面

登录后，点击左侧导航栏的 **流量管理 → HTTP 规则**。

如果页面顶部有节点选择器，先选择 `local`（本地节点）或你要使用的节点。

![HTTP 规则页面](/screenshots/panel-http-rules.png)

---

## 3. 添加规则

点击 **添加规则**，表单分为三个标签页：**基础配置**、**高级配置**、**Relay 配置**。我们从基础配置开始。

### 基础配置

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| 前端访问地址 | `app.example.com` | 选择 `http://` 或 `https://`，右侧填写用户访问的域名。 |
| 后端地址 | `http://192.168.1.100:8096` | 选择后端实际使用的协议，右侧填写后端的 IP 或域名和端口。 |

![添加 HTTP 规则](/screenshots/panel-http-rule-form.png)

下方还可以填写标签、选择出口 Profile，并确保 **启用规则** 保持开启。

### 高级配置

这里控制一些更精细的行为：

| 选项 | 说明 |
| --- | --- |
| 代理 302/307 重定向 | 开启时，后端返回的跳转地址会被改写为前端域名；关闭时原样透传。默认开启。 |
| 出口 Profile | 选择出站经过的代理（如 SOCKS / HTTP 代理、WireGuard 出口）。`Direct` 表示直连。 |
| 请求头 | 可以自定义发送给后端的请求头。 |

### Relay 配置

如果你希望流量先经过中继节点再到达后端，可以在这里添加 Relay 层。Relay 的概念详见 [L4 规则与 Relay](./l4-relay.md) 和 [Relay 参考](../reference/relay.md)。

---

## 4. 保存并启用

保持 **启用规则** 开启，点击 **创建规则**。回到列表页看到新规则出现，说明面板已保存成功。`local` 节点会通过心跳同步配置，并开始监听请求。

---

## 5. 配好 DNS 和端口

规则保存后，还需要确保外部能访问到：

- **域名解析**：入口域名已解析到节点的公网 IP。
- **防火墙放行**：安全组和系统防火墙已放行访问端口（HTTP 通常是 `80`，HTTPS 通常是 `443`）。
- **后端可达**：节点能正常访问后端服务。
- **后端权限**：如果后端需要登录、邀请码等，这些仍然由后端控制，反向代理不会绕过。

---

## 6. 启用 HTTPS（可选）

新手推荐使用 HTTP-01 自动签发证书，无需配置 Cloudflare Token：

1. 把前端访问地址改成 `https://app.example.com`
2. 确认域名已解析到节点
3. 确认节点 `80`、`443` 端口已放行
4. 保存规则，等待 Agent 同步后自动申请证书

HTTP-01 验证会临时占用 `80` 端口完成域名验证。如果节点无法开放 `80` 端口，或者需要通配符证书，请改用 DNS 验证，详见 [证书与 HTTPS](./certificates.md)。

在浏览器打开入口地址：

```text
http://app.example.com
```

如果能看到目标页面，说明规则已经生效。

## 流式恢复与重试

HTTP 代理内置了中断恢复和安全重试机制，用来改善大文件播放或下载时的偶发断流。它不会绕过后端权限，也不会让不可达的后端变得可用。

支持恢复的场景：

- `GET` 请求的完整响应
- `GET` 请求的单区间 `Range` 响应

后端需要支持 `Accept-Ranges: bytes`，并返回稳定的 `ETag` 或 `Last-Modified`。如果校验信息不一致，恢复会停止并返回错误。相关环境变量详见 [环境变量参考](../reference/environment.md)。

---

## 什么时候关闭 302/307 改写？

以下场景可以在规则中关闭 **代理 302/307 重定向**：

- CDN 回源需要保留原始重定向地址
- 多跳转链接需要客户端直接访问中间地址
- OAuth 回调需要重定向地址原样传递

## 排查清单

如果打不开，按以下顺序检查：

1. 域名是否解析到正确的节点 IP
2. 节点端口是否被防火墙放行
3. 节点是否能访问后端（在节点上执行 `curl -I <后端地址>` 测试）
4. 规则是否选择了正确的 Agent
5. 规则是否保持启用状态
6. 后端是否限制了 Host、地区、IP 或需要特殊路径

| 现象 | 常见原因 |
| --- | --- |
| 打开域名没反应 | DNS 未生效，或端口未放行 |
| 规则保存了但访问不到后端 | 节点无法访问后端，或后端协议、端口填错 |
| HTTPS 访问失败 | 还未配置证书，建议先用 HTTP 验证链路正常后再启用 HTTPS |
| 登录页能打开但播放失败 | 节点到后端线路不稳定、后端限制反代，或客户端仍在请求后端直链 |
| 代理到了错误的服务 | 前端或后端地址填错，仔细检查协议、域名、端口 |
